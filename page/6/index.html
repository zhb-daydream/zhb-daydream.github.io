自然语言处理NLP

发表于 2023-07-06 分类于 搞点课外知识
本文字数： 1.2k 阅读时长 ≈ 4 分钟

今年（2023年）年初大概3月份的时候，AI绘画和AI对话都特别火 ，前者的大新闻是 Stable-Diffusion 火出圈，后者则是 OpenAI 公司的 ChatGPT 的火出圈。照理来说，就应该追热点赶紧列一下写个博客的，结果现在都没热度了，甚至我都不怎么用了，才想起还是得补一补笔记。

NLP

AI对话本质上是AI在自然语言处理（NLP，Natural Language Processing）领域的一个新应用。NLP也是计算机科学领域、人工智能领域的重要方向，研究的是如何用自然语言（而不是二进制语言）实现人与计算机之间的有效通信。

NLP所面对的核心难题在于自然语言文本和对话广泛存在着各种各样的歧义性或多义性（ambiguity），即同一段字符串在不同的场景下会有不同的含义，虽然这正是自然语言的魅力所在，但也是计算机处理中必须消除的东西，因为机器只是机器。

抛开理论和技术不谈（我也不懂啊），NLP的应用方向主要有6种：

  • 文本挖掘：对文本进行分类、摘要、分析、信息可视化、交互式呈现
  • 信息检索：对文档词汇建立索引、对查询信息进行分析并匹配索引库
  • 句法语义分析：针对文本语句进行分词、标记、实体识别、句法分析、多义词消歧等
  • 机器翻译：根据一种自然语言自动生成另一种自然语言
  • 问答系统：利用计算机自动回答用户所提出的问题，包括问题理解、关键信息抽取、检索匹配知识库、反馈
  • 对话系统：包括用户意图识别、通用聊天引擎、问答引擎、对话管理系统，还要具备多轮对话能力

针对这6个应用方向，又可以把NLP总结为2大类核心任务：

  • 自然语言理解（NLU）：Natural Language Understanding，指计算机系统对自然语言文本进行分析、理解和推理的过程。NLU技术包括词法分析、句法分析、语义分析和语用分析等方面，旨在使计算机能够理解自然语言文本的含义和意图。NLU技术是自然语言处理的核心技术之一，它是实现智能对话、文本分类、信息抽取等应用的基础。
  • 自然语言生成（NLG）：Natural Language Generation，指计算机系统根据特定的规则和语言模型，生成自然语言文本的过程。NLG技术包括文本规划、语言生成和输出等方面，旨在使计算机能够生成符合人类语言习惯和规范的自然语言文本。NLG技术的应用包括自动摘要、自动文摘、智能写作等方面。

结构化拖延-拖着拖着就把事情做完了？

发表于 2023-07-05 分类于 搞点课外知识
本文字数： 1.5k 阅读时长 ≈ 5 分钟

拖延是大家经常会碰到的事情，一种重要的事也很可能会一直拖到临近 DDL 才会弄完，甚至可能会导致更多的事情完成不了。但斯坦福一个荣誉客座教授、哲学家约翰·佩里在《拖拉一点也无妨》里就提出了一个结构化拖延（structured procrastination）的概念，通过合理规划任务，利用拖延症来完成更多的事情。当然，作者本人提出该理论之后，花了15年才总算把这书写完，可以说很切题了。

战术拖延策略

结构化拖延这个概念并不是让人根除掉拖延的坏习惯，而是与它进行妥协，进行某种战术拖延。一般而言，拖延就是单纯地不想干某件事，而且可能越紧迫，就越不想干，比如考试前由休闲地玩手机变成焦虑地玩手机。

而结构化拖延，就是要利用“逃避”的这个特性，在拖延正事的时候，给自己安排一些没那么重要、但也是次重要的事情，从而完成掉它。而这个结构化，就是指如何去安排自己的任务清单。

首先，要构建一个优先级的任务清单：把看起来最紧急迫切的事排在最前头，一些其他值得一做但没那么紧急的事情排在后面。这个清单可以只在脑子里想想，也可以专门写下来，尤其是当这件事不能在短时间（比如几分钟）内完成的话，就应该记下来，这样可以帮助我们专注在当前的事情上，而不会一直在脑子里转着从而引起焦虑。

接着，就是对任务清单进行分解，把不能轻易完成的任务分解地足够细（甚至可以做到那种，打开教材、新建一页PPT、列出题目的详细程度）。“难事做于易，大事做于细”。这个目的是避免我们因为害怕大任务而逃避，同时也能帮助思考如何解决问题，可以不断肯定自己的进展，从而获得更多成就感而非挫败感。

最后，就是任务的执行了。只要上面的任务清单设计的足够合理，那在拖延正事的时候，就不至于无事可做而跑去用奶头乐消磨时光。这时候“拖延”的力量会迫使我们去做任务清单上一些别的事情，就像我之前为了逃避改论文而写了一堆博客文章一样。在这个拖延过程中，及时记录自己每天完成的“杂事”，久而久之，就会发现自己已经完成了不少东西，也就不容易焦虑了。毕竟，焦虑常常是因为我们觉得有太多的事情要做，却又无从下手。

别让自己无事可做

其实结构化拖延的核心，就是构建一个合理的任务清单。居于榜首的任务应当是有明确的截止期限，但拖延的后果其实又不是特别大的（比如给自己定下这个月底要把小论文改完，但就算超时了，也没有实质损害）。而排在后面的则是没那么重要，但又是有用的小事情（比如看一篇小论文并用三句话概括）。不要因为小事情“不重要”就不列上去，用更多的事情填满任务清单，是结构化拖延的重要环节，也就是让自己有事可做。

一个反例就是任务清单上只有寥寥几项任务，还全都是最重要的级别。这时候，按照拖延特性，逃避做这些重要事的唯一选择就只剩下啥也不做了，跑去玩游戏、看微信、刷短视频，慢慢地停不下来，想着明天再做也无妨，最终陷入拖延，甚至习得性无助感，任何积极的事情都不想做，包括吃饭睡觉。

拖延本身是有问题的

结构化拖延不能解决拖延问题，有时候，拖延是由某些更深层的问题导致的，这可能需要医学手段治疗，而不是一些哲学、生活策略就能解决。面对难题时，有逃避困难的心态是很正常的，因为困难会带来不适，而我们不喜欢这样的不适。当然，人与人之间不能一概而论，毅力更强的人能够忍受更大的不适感。因为不适，就会逃避，就会拖延。

结构化拖延可以通过充分利用拖延过程来完成一些杂事，提高自己的成就感，减少内疚和绝望，从而帮助增强行动力。而如果自己拖延到，连重要事以外的东西都不想做，那也许就不是这个策略能解决的问题了，或许该散散心、换个环境了。

Games Workshop公司

GW（一个英国桌游厂商，Games Workshop）一共基于两个战锤世界观设计了对应的桌游，也是战锤IP的持有者。

  1. 中古战锤（Warhammer Fantasy）：以中世纪为背景，问世于1983年
  2. 战锤40K（Warhammer 40K）：以未来的星际战争为背景，问世于1987年

GW 这个公司最初由几位桌游爱好者创立，获得过人气桌游龙与地下城（Dungeons & Dragons，在《生活大爆炸》中出现过）在英国的代理权，后来还推出了指环王的官方授权桌游。随后就逐渐将重心转移到打造自己的 IP 产品线上，并催生出不少佳作，包括走奇幻风的“中古战锤”（Warhammer Fantasy）系列桌游和走太空歌剧风的“战锤40K”（Warhammer 40K）系列桌游。这两个系列桌游的最典型差异就在于，“中古战锤”系列桌游的玩法侧重大兵团作战，而“战锤40K”系列桌游里的战斗多基于小队制。

战锤IP

中古战锤与战锤40K系列桌游里的世界观、游戏背景慢慢地就变成一种IP（类似于西游记、三国一样），由 GW 授权给其他游戏工作室进行相应的游戏开发。不过 GW 近十年来大批量地授权IP，导致市面上打着“战锤”名号的电子游戏特别多，但是 steam 上就有46款玩法不同的“战锤”游戏。以下只列部分名气比较大的战锤系列游戏。

什么是Packgage.json?

package.json 文件是用来管理本地安装 npm 包的唯一文件。它描述了当前项目依赖的包的列表，包含项目依赖包的版本，便于项目管理和移植，格式是严格的JSON格式。

依赖包

在 package.json 文件中指定项目依赖的包，这样复制项目时，可以使用 npm install 下载

依赖方式

  • dependencies：在生产环境中需要用到的依赖

  • devDependencies：在开发、测试环境中用到的依赖，在上线打包后的代码是不存在的

版本选择

  • "package" : "1.2.x"会匹配最性的小版本依赖包
  • "package" : "1.x"会匹配最新的大版本依赖包
  • "package" : "~1.2.6"会匹配最近的可用版本依赖包，这里的~表示约等于

版本号规范

项目的版本号都是从 1.0.0 版本开始。以后要升级版本应该遵循以下标准：

  • 补丁版本：解决了 Bug 或者一些较小的更改，增加最后一位数字，比如 1.0.1
  • 小版本：增加了新特性，同时不会影响之前的版本，增加中间一位数字，比如 1.1.0
  • 大版本：大改版，无法兼容之前的，增加第一位数字，比如 2.0.0

在 package.json 中最重要的属性是 name 和 version 两个属性，这两个属性是必须要有的，否则模块就无法被安装，这两个属性一起形成了一个 npm 模块的唯一标识符。模块中内容变更的同时，模块版本也应该一起变化。

name属性就是你的模块名称，下面是一些命名规则：

  • name必须小于等于214个字节，包括前缀名称在内（如 xxx/xxxmodule）
  • name不能以"_“或”."开头 不能含有大写字母
  • name会成为url的一部分，不能含有url非法字符，不要使用和node核心模块一样的名称，name中也不要含有"js"和"node"
  • name属性会成为模块url、命令行中的一个参数或者一个文件夹名称，任何非url安全的字符在name中都不能使用，也不能以"_“或”."开头
  • name属性也许会被写在require()的参数中，所以最好取个简短而语义化的值
  • 创建一个模块前可以先到npm的网站（https://www.npmjs.com/name）查查name是否已经被占用
  • name属性可以有一些前缀如@myorg/mypackage

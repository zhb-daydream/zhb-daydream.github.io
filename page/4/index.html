什么是Packgage.json?

package.json 文件是用来管理本地安装 npm 包的唯一文件。它描述了当前项目依赖的包的列表，包含项目依赖包的版本，便于项目管理和移植，格式是严格的JSON格式。

依赖包

在 package.json 文件中指定项目依赖的包，这样复制项目时，可以使用 npm install 下载

依赖方式

  • dependencies：在生产环境中需要用到的依赖

  • devDependencies：在开发、测试环境中用到的依赖，在上线打包后的代码是不存在的

版本选择

  • "package" : "1.2.x"会匹配最性的小版本依赖包
  • "package" : "1.x"会匹配最新的大版本依赖包
  • "package" : "~1.2.6"会匹配最近的可用版本依赖包，这里的~表示约等于

版本号规范

项目的版本号都是从 1.0.0 版本开始。以后要升级版本应该遵循以下标准：

  • 补丁版本：解决了 Bug 或者一些较小的更改，增加最后一位数字，比如 1.0.1
  • 小版本：增加了新特性，同时不会影响之前的版本，增加中间一位数字，比如 1.1.0
  • 大版本：大改版，无法兼容之前的，增加第一位数字，比如 2.0.0

安装参数 --save 和 --save -dev

添加依赖时可以手动修改 package.json 文件，具体而言，添加或者修改文件中的 dependencies devDependencies 中的内容即可。另一种方式是用命令行，在使用 npm install 时增加 --save 或者 --save -dev 后缀：

在 package.json 中最重要的属性是 name 和 version 两个属性，这两个属性是必须要有的，否则模块就无法被安装，这两个属性一起形成了一个 npm 模块的唯一标识符。模块中内容变更的同时，模块版本也应该一起变化。

name属性就是你的模块名称，下面是一些命名规则：

  • name必须小于等于214个字节，包括前缀名称在内（如 xxx/xxxmodule）
  • name不能以"_“或”."开头 不能含有大写字母
  • name会成为url的一部分，不能含有url非法字符，不要使用和node核心模块一样的名称，name中也不要含有"js"和"node"
  • name属性会成为模块url、命令行中的一个参数或者一个文件夹名称，任何非url安全的字符在name中都不能使用，也不能以"_“或”."开头
  • name属性也许会被写在require()的参数中，所以最好取个简短而语义化的值
  • 创建一个模块前可以先到npm的网站（https://www.npmjs.com/name）查查name是否已经被占用
  • name属性可以有一些前缀如@myorg/mypackage

基本操作

Hexo 的 NeXT 主题自带有 Mermaid 渲染器，只要先装个 Mermaid 的渲染插件hexo-filter-mermaid-diagrams，然后在 NeXT 主题根目录的配置文件 _config.yml 中启用 Mermaid 即可。

报错

本来是一件很简单的事情，但偏偏出岔子了！

首先，一个有效的检查方法，就是去看 Hexo 生成的 public 目录下存放的各 .md 文件生成的 index.html，所有 Mermaid 生成的流程图都应该是包含在一个标签类里的，这时候如果流程图没有class = “mermaid”的字段，那就是第一步安装的hexo-filter-mermaid-diagrams插件没有解析成功，这时候就要检查安装是否成功、依赖库是否正确。

其次，在本次安装hexo-filter-mermaid-diagrams插件时，就已经报错了，再次输入npm audit，可以看到文本中的大意是说，一个叫optionator的库所依赖的另一个word-wrap的库有问题，处于vulnerable versions中。

在word-wrap的github库下的#33讨论中，可以大致判断出是该库的一些正则表达式（regex）可能存在bug，而该库所有者Jon Schlinkert在2023.5月的时候出车祸了，导致后续的PR（Pull Request）至今无人merge，很多人也在吐槽原作者根本就无意继续维护这库，这在JS（JavaScript）社区是个耻辱，因为这依赖库很常用，每周有2.9千万次的下载量。为此，aashutoshrathi自行fork了一下目前的库，修改了bug后发行了自己版本的word-wrap，用于临时替换不再维护的版本（someone brave enough can publish a version of the package that can be a replacement）。

注：git中的各个动词举例如下：

  • 你做的试卷就像仓库，你的试卷肯定会有很多错误，就相当于程序里的 bug。
  • 老师把你的试卷拿过来，相当于先 fork。
  • 在你的卷子上做一些修改批注，相当于 git commit。
  • 最后把改好的试卷给你，相当于发 pull request.
  • 你拿到试卷重新改正错误，相当于 merge。

更换依赖库

接下来的问题就是更换一个能用的word-wrap版本了，也是参考的#33讨论。

首先在 Hexo 根目录里打开 cmd，安装aashutoshrathi依赖库：npm install --save @aashutoshrathi/word-wrap，随后就能在 Hexo 根目录的 node_modules 文件夹里看到有 node_modules\@aashutoshrathi\word-wrap。

接下来，就是在 Hexo 根目录下的 package.json 文件（指定当前项目所依赖的包）里进行依赖库的替换（由于仓库名等问题，不能直接把aashutoshrathi的word-wrap里的文件直接替换掉原版本文件）。在末尾处加上如下的 overrides 命令，将默认的 word-wrap 库替换成 @aashutoshrathi/word-wrap 的版本。

什么是 Mermaid

在 Markdown 的文本编辑器 Typora 中，我们可以通过 Mermaid 来实现流程图、甘特图、时序图等图像的绘画，来明确信息之间的关系。这里的 Mermaid 实际上是一种基于 Javascript 的绘图工具/脚本语言，使用类似于 Markdown 的语法，让用户可以方便快捷地通过代码创建图表。

与 Markdown 语言类似，为了使用 Mermaid，就需要能支持 Mermaid 渲染功能的编辑器，Typora 便是其中一个。在插入代码块后，将代码块的语言选择为 Mermaid 便能进行渲染了。

Mermaid 支持的几种类型图：

  • 流程图：使用graph关键字
  • 饼状图：使用pie关键字
  • 序列图：使用sequenceDiagram关键字
  • 甘特图：使用gantt关键字
  • 类图：使用classDiagram关键字
  • 状态图：使用stateDiagram关键字
  • 用户旅程图：使用journey关键字

实例：朱元璋家谱简图，圆圈代表皇帝。

graph LR
emperor((朱八八))-.子.->朱五四-.子.->朱四九-.子.->朱百六
朱雄英--长子-->朱标--长子-->emperor
emperor2((朱允炆))--次子-->朱标
朱樉--次子-->emperor
朱棡--三子-->emperor
emperor3((朱棣))--四子-->emperor
emperor4((朱高炽))--长子-->emperor3

流程图

方向

在开头处声明流程图的方向

  • graph或graph TB或graph TD：从上往下
  • graph BT：从下往上
  • graph LR：从左往右
  • graph RL：从右往左

节点形状

  • 无名字的结点：直接写内容，此时结点边框为方形；节点内容不支持空格
  • 有名字的结点：节点名后书写内容，内容左右有特定符号，结点边框由符号决定；节点内容可以有空格

下面实例中，id1-id6是节点名，可随意定义

graph
    默认方形
    id1[方形]
    id2(圆边矩形)
    id3([体育场形])
    id4[[子程序形]]
    id5[(圆柱形)]
    id6((圆形))
graph
    id1{菱形}
    id2
    id3[/平行四边形/]
    id4>非对称形]
    id5[/梯形\]
    id6[\反向梯形/]

连线样式

（1）实箭头：无文本箭头（1种写法）、有文本箭头（2种写法）

graph LR
a-->b--文本1-->c-->|文本2|d

（2）粗箭头：无文本箭头（1种写法）、有文本箭头（1种写法）

graph LR
a==>b==文本==>c

（3）虚线箭头：无文本箭头（1种写法）、有文本箭头（1种写法）

graph LR
a-.->b-.文本.->c

（4）无箭头线：将以上情况中的>删去

graph LR
a---b
b--文本1!---c
c---|文本2|d
d===e
e==文本3===f
f-.-g
g-.文本.-h

（5）延长连线：增加相应字符即可，如下图中的B到E，连线中增加了一个-。字符可多次添加。

graph LR
    A[Start] --> B{Is it?};
    B -->|Yes| C[OK];
    C --> D[Rethink];
    D --> B;
    B --->|No| E[End];

连线形式

（1）直链：和上面一样

graph LR
   A -- text --> B -- text2 --> C

（2）多重链：可以使用&字符，或一个一个地描述

graph 
   a --> b & c--> d
   
   A & B--> C & D
   
    X --> M
    X --> N
    Y --> M
    Y --> N

子图

在子图开始的部分加上subgraph [name]，在子图结束部分加上end即可。

graph TB
    c1-->a2
    subgraph one
    a1-->a2
    end
    subgraph two
    b1-->b2
    end
    subgraph three
    c1-->c2
    end

注释

使用%%表示该行文本为注释，不会进行渲染

注意力集中之心流状态

发表于 2023-07-03 更新于 2023-07-05 分类于 搞点课外知识
本文字数： 1.5k 阅读时长 ≈ 5 分钟

刷知乎的时候刷到一个问题《为什么很多父母觉得所有游戏，尤其是moba类可以随时暂停？》下的一个回答，感觉挺有意思，也是在里面接触到了心流状态这个概念。大概就是讲答主小时候很容易就能进入注意力高度集中的状态，比如小学六年级的时候思考一道复杂的奥数题想了一个多小时，以至于他妈妈拍他的时候他会被猛的下一跳。答主在多次进入这种“视外界为无物”的状态时被打断，心情也慢慢变得暴躁。后来干什么事都“眼观六路，耳听八方”，根本没法把全部心神投入进某件事，无论是做题还是娱乐都不沉浸了，以至于患上躁郁症、没法集中注意力、经常走神、记忆力越来越差。

想起来我以前小学的时候，也是试过有一节自习课，计算机老师就给每个人布置个比较超纲的数学题（好像是等比数列还是等差数列？），目的也不是让大伙做，估计就是某种社会学测试，然后我确确实实地从上课一直思考到下课，期间也没有走神或者玩，就是在想，尽管没有想出来，但那种状态倒是印象深刻的注意力高度沉浸的状态。后来抑郁症那会，确实完全没法集中注意力，只要稍微安静一点，脑子里就会有好几个人在自言自语，以至于我得一直戴着耳机调高音量（大概这也是为什么现在听力下降了吧）。

有点偏题了，总之这个心流状态（Flow model）的概念，最早是由心理学家米哈里·契克森米哈赖（Mihaly Csikszentmihalyi）在1990年的《心流：最优心理学体验》（Flow: The Psychology of Optimal Experience）里提出来的，其初衷是为了研究为什么人类并没有因为信息技术爆发等重大变革事件而更幸福。

米哈里将心流（flow）定义为一种将个人精神力完全投注在某种活动上的感觉，心流产生时会伴有高度的兴奋及充实感，他还认为心流：“是一种几乎自动的、不须花费力气的、但又高度集中的感觉状态”。

如下图所示，目前心理学界认为，只有当个人的技巧水平（横坐标）以及所做事情的挑战水平（纵坐标）都处于较高水平的时候，才会出现心流状态，而如果两者都是较低水平，这时候人所处的状态实际上是属于冷漠、无感。对应于刚开始学习一项运动的时候会感觉没劲，而等熟悉技巧以后，用更高级的技能去挑战更难的技术动作时就会感觉很沉浸、投入。

稍微完整一点而言，当一个人能力明显低于完成所做任务时，他会感觉到焦虑；而一个人能力远高于任务所需时，他会感觉到无聊；只有当能力与任务难度刚好匹配是，他才不会焦虑或者无聊，才可能产生心流，也就是所谓的心流状态。

对个人而言，一个好的工作状态应当有这种特点，以使得自己处于比较舒服的心理环境：

  • 有一段完全不被打扰的时间和空间
  • 有明确的工作目标和完成条件
  • 工作内容存在挑战性，但不是完全实现不了
  • 有实时奖励反馈可以提高成就感
  • 根据阶段性的工作完成度去制定个人能力的发展方向

而除了自我提升以外，在管理、游戏设计、客户体验等领域也同样可以稍微利用这种心流状态的概念，比如就游戏设计而言，大概有4大特征可以让玩家快速地进入心流状态：

  • 目标：玩家只要努力就可以达成具体的结果。比如，装备的等级限制，技能的等级限制，也就是很多游戏为什么能够让你获得高级的装备但是当下却使用不了的原因。
  • 规则：为玩家实现目标提供可行的途径。做出规则限制，时间限制，提高玩家的专注度。
  • 反馈：告诉玩家距离目标还有多远。通过量化的经验条等反馈系统来让玩家清晰的感受到距离升级还有多远。
  • 参与：把高挑战的工作视为安全且愉快的事情。让玩家认为游戏过程对自己很有意义和帮助，因而投入的精力也就会让自己开心，形成正反馈。

今天开始，决定“个人吐槽”这个分类正式改为“日记”了。虽说正经人谁写日记，之前取名“个人吐槽”也是为了让这些文笔不那么正式，但想了想，写这些就是为了趁自己还记得的时候及时记录下此刻的心情，就像QQ的说说或者微信的朋友圈一样，那这不就是日记嘛。而且这博客主要就是给自己看的，哪还用管什么正经人不正经人的。

昨晚临睡前上 JMR 的投稿系统查，发现自己投的那篇论文已经 decision in process 了，才10天！这大概就是编辑直接拒了，甚至都没有送外审（之前 medical physics 编辑拒都用了20天）。虽然口上说着没事没事，赶紧拒了赶紧投下一个，但心里还是难受，一直到2点才睡着，5点醒了一次，又迷迷糊糊睡到8点，然后就睡不着了。期间各种做噩梦，还梦见编辑写了封信骂我的工作有多差劲。唉，只能说抗压能力还不够，如果这都破防，那以后的博士生涯也不用混了，还是得多碰碰这种壁，才能锤出大心脏。就像我第一次玩PVP游戏，好像是龙之谷还是剑灵，看着对手冲我跑过来的时候就心跳特别快，现在打王者输多了之后，好像也习惯了不少。

今天的一大件事就是把东区太湖新村租的房子退了，搬空了东西，又跑了5趟才把所有行李搬回宿舍。搬空房间的那一刻感觉特别恍惚，三年前本科毕业，因为疫情不能去美国读博而gap了一年，暂时留在科大做科研助理，才租了这里。呆着呆着，莫名其妙决定就留在国内读博了，然后一晃三年过去，没有论文、没有课题（也算有一个，但也只是刚开始），身边的硕士同学纷纷毕业，剩我一个人留在这城市，真是奇妙啊。三年，不短不长的时间，就像初中、高中一样，一晃就过去，世界就变了样。三年之后我就是博四，而博五开始学校就不提供宿舍，到时候不管能不能毕业，又是一轮搬空房间、去新的居住点，到时候的光景又是如何？我还是精神正常的吗？这种未来真是不敢深究呢。
